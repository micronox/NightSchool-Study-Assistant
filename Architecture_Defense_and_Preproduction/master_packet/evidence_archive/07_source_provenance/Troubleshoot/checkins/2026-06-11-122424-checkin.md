## Fresh Evidence

### Apollo config and log history disagree about Telegram ownership over time

Current Apollo environment comments indicate:

- Apollo is API-only by default
- Commander owns the shared Telegram bot lane
- a separate Apollo Telegram bot would need to be added later if desired

This is highly relevant because Apollo's historical logs clearly show Apollo
did try to connect to Telegram and did experience polling conflicts.

Interpretation:

- the intended ownership model changed over time
- or the runtime drifted away from the intended ownership model
- or both

This is one of the best concrete examples yet of doctrine/config drift becoming
runtime failure.

### Apollo config now points to `8646` while old runtime evidence hit `8645`

Fresh config evidence:

- Apollo `config.yaml` currently specifies API server port `8646`

Historical runtime evidence:

- Apollo `gateway.log` repeatedly failed on `8645 already in use`

Interpretation:

- at least one meaningful port change happened between the older broken runtime
  and the current saved configuration
- stale docs and stale state are not the only issue; the persisted profile
  config also evolved while the system was unstable

This makes it harder to trust any single artifact from the old stack unless it
is tied to a specific timestamp.

### Local API probing confusion has older precedent too

`L:\WSL\runtime\hermes-api-bridge.log` shows a long history of local

The local stack appears to have gone through multiple incompatible API surface

- route exists vs route missing
- open probe vs authenticated probe
- bridge expectations vs direct profile API expectations

## New Conclusions Since Last Check-In

1. Telegram lane ownership between Commander and Apollo drifted over time in a
   way that directly matches the polling-conflict history.
2. Apollo's API port also drifted over time (`8645` in broken runtime evidence,
   `8646` in current config), which further weakens trust in stale profile
   state.
3. Local health-check/integration callers have used inconsistent API
   expectations over time, from 404-era bridge probing to invalid-key probing.
4. The control-plane break is increasingly explained by moving ownership rules,
   partial maintenance, and incompatible assumptions rather than one single
   mechanical failure.

## Current Root-Cause Ranking

1. Ownership drift between documented roles and actual runtime surfaces
2. Shared runtime drift plus dirty update/launch lifecycle in `hermes-agent`
3. Scheduler/launcher ownership failure for profile gateways
4. Stale profile runtime truth (`gateway.pid` / `gateway_state.json`)
5. Duplicate gateway/control-plane copies and Telegram polling conflicts
6. Recursive workspace duplication in `site-studio`
7. Browser `/chat` session reuse through `claude-last-session`
8. Inconsistent local API probing / bridge expectations
9. OpenRouter throttling amplified by orchestration behavior
10. GPU/driver issue as a secondary concern rather than the leading cause
