profile wrappers.

- the shared runtime tree itself was not rewritten in the exact June 10 failure

1. shared runtime drift / update churn

## New Conclusions Since Last Check-In

1. The June 10 failure window does not look like a clean full reinstall of the
   shared `hermes-agent` tree.
2. The stack still shows a pattern of updating the shared runtime while live
   gateway processes were blocking file replacement.
3. Partial maintenance/update drift is now a stronger suspect than a single
   catastrophic rewrite.
4. Local callers were also misaligned with Commander API authentication, which
   means the control plane was not only unstable but also inconsistently
   integrated.
5. The system repeatedly tolerated half-alive conditions instead of enforcing a
   clean fail/stop, which likely compounded every other issue.

## Current Root-Cause Ranking

1. Shared runtime drift plus dirty update/launch lifecycle in `hermes-agent`
2. Scheduler/launcher ownership failure for profile gateways
3. Stale profile runtime truth (`gateway.pid` / `gateway_state.json`)
4. Duplicate gateway/control-plane copies and Telegram polling conflicts
5. Recursive workspace duplication in `site-studio`
6. Browser `/chat` session reuse through `claude-last-session`
7. Misconfigured local API probing / invalid auth expectations
8. OpenRouter throttling amplified by orchestration behavior
9. GPU/driver issue as a secondary concern rather than the leading cause
