## Summary

This pass strengthens the scheduler/control-plane theory. The Hermes scheduled
gateway tasks still appear to be the intended owners, but they all share the
same failure timestamp and the same failure code, while the actual gateway
processes are dead and the profile state files still claim healthy runtime.

That pattern looks less like isolated service crashes and more like a broken
shared launch/ownership path.

## Fresh Evidence

### Scheduled tasks still show synchronized failure

Hermes gateway tasks observed:

- `Hermes_Gateway_commander` -> `Ready`
- `Hermes_Gateway_apollo` -> `Ready`
- multiple adjacent profile tasks -> `Disabled`

Common failure shape:

- `LastRunTime`: `2026-06-10 01:19:50 AM`
- `LastTaskResult`: `4294967295`
- `NextRunTime`: blank

Interpretation:

- the stack still believes these scheduled tasks are the owning control-plane
  surfaces
- they all failed in the same window
- none appear to have healthy recurring ownership after that point

This is strong evidence of a centralized break in the launch path or shared
runtime assumptions.

### Expected gateway-service wrapper path does not appear healthy

I attempted to read:

- `$USER_HOME\.hermes\profiles\commander\gateway-service\run-gateway.cmd`
- `$USER_HOME\.hermes\profiles\apollo\gateway-service\run-gateway.cmd`

No usable output came back from those paths during this pass.

That suggests at least one of:

- the wrapper file is missing
- the wrapper filename/path is different from the expected convention
- the gateway-service directory exists but is not the true runtime owner anymore

This matters because it weakens trust in the profile-local launch story and
supports the idea that ownership drifted away from the documented structure.

### Dead gateway PIDs remain absent as live processes

No live process exists for the profile-recorded gateway PIDs:

- Commander stale PID: `30068`
- Apollo stale PID: `30044`

This remains consistent with the earlier finding that the profile JSON state is
advertising a live runtime that does not exist.

### Config drift between profiles is still small at the top level

Both Commander and Apollo `config.yaml` files still show the same broad model
posture:

- provider: `openrouter`
- model: `nvidia/nemotron-3-super-120b-a12b`
- long context length
- local terminal backend
- compression enabled

This means the break is not obviously explained by radically different top-level
model/runtime policy between the two profiles. The stronger explanation is still
launch ownership, stale state, or platform-specific sub-configuration.

## New Conclusions Since Last Check-In

1. The scheduled-task layer is likely part of the root failure, not just a
   symptom, because the gateway tasks failed in the same time window with the
   same result code.
2. The legacy stack still presents “Ready” scheduled owners without any healthy
   corresponding live control-plane ports.
3. The missing or opaque `gateway-service` wrapper path makes the effective
   launch chain less trustworthy than the folder layout implies.
4. The profile configs are similar enough that the control-plane collapse still
   looks more like orchestration drift than model-choice divergence.

## Current Root-Cause Ranking

1. Broken shared launch path / scheduler ownership for Hermes gateways
2. Stale profile runtime truth (`gateway.pid` / `gateway_state.json`)
3. Duplicate gateway/control-plane copies and Telegram polling conflicts
4. Recursive workspace duplication in `site-studio`
5. Sticky session reuse / poor session lifecycle normalization
6. OpenRouter throttling amplified by orchestration behavior
7. GPU/driver issue as a secondary concern rather than the main break
