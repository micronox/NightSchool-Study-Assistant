# WSL Stack Check-In

- Timestamp: 2026-06-11T13:54:21.526Z
- Scope: Hermes startup wiring, duplicate gateway evidence, and Site Studio recursive import path

## Summary

This pass strengthened two root-cause candidates significantly:

1. duplicate Hermes gateway/control-plane copies are not theoretical; the logs explicitly show repeated Telegram polling conflicts caused by another live `getUpdates` session
2. Site Studio import logic appears to have no guard against importing a folder that already contains prior managed project copies, which makes the recursive `WSL\site-studio\projects\project...\WSL\site-studio\projects\project...` tree much more believable as a real self-ingestion bug

The stack split is now partly explained as intentional and partly explained as drift:

- intentional: `hermes-up.ps1` now treats Hermes as a DeepSeek/OpenRouter control plane and treats local Gemma as a separate lane
- drift/failure: the expected Hermes ports are still down right now despite prior successful startups in logs

## New Evidence Gathered

### Hermes startup is intentionally split from local Gemma

`L:\WSL\hermes-up.ps1` does not restore the old local Gemma lane. It calls:

- `L:\WSL\hermes-deepseek-openrouter-up.ps1`

That script:

- pins Hermes to `deepseek/deepseek-v4-pro`
- sets provider to `openrouter`
- expects Hermes API on `:8642`
- explicitly says local Gemma remains a separate lane

This means part of the “split brain” is by design:

- Hermes control plane = OpenRouter/DeepSeek seat
- local Gemma/Open WebUI = separate utility/worker lane

That said, the current machine state still shows the Hermes side is not actually live right now.

### Hermes bridge depends on a live main Hermes API

`L:\WSL\hermes-bridge-up.ps1` only starts if:

- `http://127.0.0.1:<MainPort>/v1/models` is reachable first

So if `:8642` is down, `:8643` must also stay down. This explains why the bridge is absent today without needing a second independent failure.

### Duplicate Telegram polling is directly proven

This is the biggest new finding of the pass.

`$USER_HOME\.hermes\profiles\commander\logs\errors.log` and `gateway.log` contain repeated lines like:

- Telegram polling conflict
- “terminated by other getUpdates request”
- “make sure that only one bot instance is running”

These conflicts repeat for long stretches on 2026-06-06. This is direct evidence that duplicate gateway copies or overlapping bot sessions existed.

That aligns with:

- the documentation warning in `STARTUP-AND-SHUTDOWN.md`
- your observation that chatter and odd bursts were happening

It also makes the Telegram/OpenRouter burst hypothesis much stronger. If two control-plane copies were alive, they could both react to events or cron surfaces.

### Gateway startup history shows many launches

`gateway-exit-diag.log` records numerous `gateway.start` events across multiple days and PIDs, including both:

- `hermes ... gateway run`
- `hermes_cli.main.py gateway run`

This does not by itself prove simultaneous overlap every time, but it does show frequent manual or repeated bring-up attempts.

It increases the odds that stale sessions and duplicate polling were caused by repeated starts rather than a single clean service lifecycle.

### API server port drift is real in logs

Earlier gateway logs show:

- `:8642` used on 2026-06-06
- `:8645` used on 2026-06-09 and 2026-06-10

So the control-plane port changed across runs. That means the docs and the current gateway config/runtime were not stable over time.

This helps explain why some surfaces feel “wrong” even when the stack was previously healthy.

### Site Studio import logic has no obvious recursion guard

Relevant code in `L:\WSL\site-studio\app.py`:

- `import_project(...)` normalizes the source
- computes a destination under `PROJECTS_DIR`
- calls `copy_project_tree(source, dest)`

`copy_project_tree` uses `shutil.copytree(...)` with an ignore list based only on `SKIP_DIRS`.

Important detail:

- there is no visible guard here preventing import of a folder that already contains `site-studio\projects`
- there is no visible check that rejects a source rooted inside an existing managed project tree
- there is no visible path ancestry check blocking source trees that already embed prior `WSL` copies

That does not prove the exact trigger yet, but it means the recursive `project-daf03eb0...` tree is technically plausible from the import design.
