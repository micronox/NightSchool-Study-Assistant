# Nice-to-Haves and MVP Prep

> **Purpose:** capture the upgrades that usually make Hermes feel dramatically more useful once the core stack is stable.

**Goal:** prepare the first MVP so it can grow into a durable Hermes OS without losing history or momentum.

**Architecture:** once the mandatory stack is stable, layer in automation, observability, memory hygiene, and better Git/version discipline. These are the niceties that tend to change the workflow by a factor of 10.

**Source basis:** `Perplexity_Hermes-on-WSL Stack Analysis and Agentic OS Buildout Plan.md`, `Perplex_2_Designing a Hermes Mission Control for Multiple Workflows.md`, `Perplex_3_memory-obsidian-honcho-openviking.md`, `Perplex_4_client-work-upwork-github-isolation.md`, `Perplex_5_github-youtube-customization-routes.md`, `Perplex_6_skills-user-groups-and-community-patterns.md`.

---

## 1. Nice-to-Haves That Change the Workflow

These are the recurring “this made Hermes 10x better” items from the research:

- daily reports
- nightly summaries
- cron jobs
- repo maps
- memory hygiene
- better skill libraries
- Kanban + profiles together
- dashboard visibility
- webhook or notification hooks
- client-safe namespaces
- clean Git branching and tagging

## 2. Automation to Add After MVP

- nightly review of sessions and logs
- daily status note generation
- stuck-task detection
- repo health checks
- memory pruning or summarization
- vault sync jobs
- project-specific progress summaries
- skill improvement suggestions

## 3. Git and History Prep

If you want to preserve the evolution of the stack and be able to revisit it later:

- use frequent commits
- tag major milestones
- keep branches named by purpose
- keep generated artifacts in predictable places
- avoid mixing unrelated work into one commit
- keep the Hermes operating docs under version control

That gives you a clean time-travel trail later.

## 4. Memory and Knowledge Hygiene

Useful niceties here are:

- automatic summaries of long sessions
- project retrospectives
- decision logs
- reusable procedure notes
- periodic cleanup of stale memory
- vault pages for architecture decisions and workflow changes

## 5. Client-Work Prep

Before serious client work scales up, add:

- client-specific profiles when warranted
- local-first provider defaults for sensitive work
- isolated vault trees
- isolated repo trees
- a safety checklist before commits or pushes

## 6. Hermes Dreaming Niceties

If Hermes Dreaming is driving the stack, the nicest upgrades are:

- auto-synthesized architecture notes
- automatic skill suggestions from repeated work
- project-level decision extraction
- task decomposition into small/medium/large items
- internal “what changed this week” reports

## 7. MVP Prep Checklist

The MVP should be ready to:

- run on WSL as authority
- attach Desktop as a client
- remember important global rules
- read and write vault notes
- run at least a handful of reusable skills
- separate client work from personal/product work
- keep a clean change history
